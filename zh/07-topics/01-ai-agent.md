# 专题1 AI Agent：让 AI 从“会回答”走向“会办事”

> 如果大语言模型像一个很会聊天的大脑，AI Agent 就是给它接上眼睛、手和工作台。它不只是回答“应该怎么做”，而是能按目标拆步骤、查资料、用工具、检查结果，直到把事情往前推进。

## 先从一个真实感强的场景说起

普通聊天 AI 像一个聪明顾问。你问：“下周去杭州开会，帮我安排一下。”它能给你一份建议：订票、看酒店、查天气、准备会议资料。

Agent 更像一个有工具权限的助理。它会先问清楚预算和时间，查高铁或机票，比较酒店位置，打开日历避开冲突，把会议资料整理成清单，必要时把待确认项交还给你。

所以，Agent 的关键不在于“更会聊天”，而在于它多了三个东西：

- **目标感**：知道现在不是随便回答，而是要完成一件事。
- **行动能力**：能调用搜索、代码、数据库、文档、邮件、日历等工具。
- **反馈循环**：每做一步都看结果，再决定下一步怎么走。

一句话概括：

> **AI Agent = 大模型 + 任务规划 + 工具调用 + 结果反馈 + 必要的人类确认。**

## Agent 和普通对话 AI 有什么区别？

| 对比项 | 普通对话 AI | AI Agent |
| --- | --- | --- |
| 你给它什么 | 一个问题 | 一个目标 |
| 它主要产出什么 | 一段回答 | 一串行动后的结果 |
| 处理方式 | 直接生成内容 | 规划、执行、观察、调整 |
| 依赖能力 | 语言理解和生成 | 语言能力 + 工具能力 + 流程控制 |
| 最大风险 | 答错、编造 | 做错、越权、把错误传下去 |

![AI Agent 与普通对话对比](../assets/images/ai-agent-vs-chat.svg)

举个很简单的判断标准：如果一个系统只是告诉你“你可以怎么做”，它更像聊天 AI；如果它能在授权范围内真的去查、去算、去改、去提交，它才更接近 Agent。

## Agent 的核心循环：想一下，做一步，看结果

Agent 完成任务时，通常不是一次性把所有步骤写死，而是反复经历一个循环：

![AI Agent 工作流程](../assets/images/ai-agent-workflow.svg)

```text
用户给出目标
    ↓
思考：现在最该做哪一步？
    ↓
行动：调用工具或生成内容
    ↓
观察：读取工具返回的结果
    ↓
判断：完成了吗？有偏差吗？
    ↓
继续、修正，或交给人确认
```

这个循环看起来简单，却是 Agent 和“普通自动化脚本”的分水岭。脚本通常按固定流程执行；Agent 会根据中途结果改计划。比如订票失败，它可以换日期、换交通方式，或者把“价格超预算”反馈给你决定。

## 三个关键能力

### 1. 规划：把模糊目标拆成可执行步骤

人类给出的目标经常很粗糙，比如：

> “帮我分析一下这个月公众号数据，看看下个月该写什么。”

一个 Agent 不能直接开写，它要先把问题拆开：

1. 读取这个月文章数据。
2. 找出阅读量、完读率、分享率较高的文章。
3. 对比不同主题和标题形式。
4. 总结可能有效的选题方向。
5. 生成下个月选题表，并标出理由。

规划能力越强，Agent 越像一个靠谱同事；规划能力弱，它就会像“很勤快但没方向的人”，忙半天也不一定忙到点上。

### 2. 工具调用：让模型连接真实世界

大模型靠训练数据知道很多事，但它的记忆不是实时数据库。工具调用让 Agent 能获得“现在”的信息，也能真正操作外部系统。

常见工具包括：

- **搜索和浏览器**：查最新资料、打开网页、核对来源。
- **代码执行器**：做计算、清洗数据、生成图表、批量处理文件。
- **数据库和知识库**：检索企业内部文档、订单、客户记录。
- **办公工具**：读写文档、整理表格、创建日程、草拟邮件。
- **业务 API**：查询库存、创建工单、发起审批、生成报告。

工具越强，责任也越大。一个能读资料的 Agent 出错，最多是答错；一个能发邮件、改数据库、提交代码的 Agent 出错，影响就会真实发生。

### 3. 观察与校验：别只会往前冲

很多 Agent 翻车，不是因为它第一步就完全错了，而是因为它没有认真检查每一步结果。

比如让 Agent “整理一份竞品分析”。它搜索到 5 个网页后，如果不检查发布日期、来源可信度、是否是广告稿，就可能把过时或带偏见的信息当成事实。好的 Agent 应该会做这些事：

- 看工具是否真的执行成功。
- 检查返回内容是否和目标相关。
- 发现证据不足时继续查，而不是硬编。
- 重要操作前停下来，请人确认。

你可以把“观察”理解成 Agent 的刹车系统。没有它，行动能力越强，风险越大。

## 哪些任务适合交给 Agent？

Agent 特别适合“目标清楚、步骤较多、需要工具、允许中途调整”的任务。

| 适合场景 | 为什么适合 |
| --- | --- |
| 编程协作 | 能读代码、改文件、跑测试、根据错误继续修 |
| 数据分析 | 能读取表格、清洗数据、画图、解释异常 |
| 研究整理 | 能搜索资料、比较来源、生成结构化摘要 |
| 客服和运营 | 能查订单、套用规则、处理重复流程 |
| 个人助理 | 能整理日程、生成清单、提醒待办 |

但有些任务暂时不适合完全放手：

- 后果严重、不可轻易撤销的操作，比如付款、删库、群发邮件。
- 规则含糊、价值判断很重的事情，比如裁员建议、医疗诊断。
- 需要真实身份责任的决策，比如签合同、审批财务。

更稳妥的做法是：**让 Agent 做准备工作和低风险操作，把关键决策留给人。**

## 设计一个 Agent 时，要先问五个问题

如果你以后要判断一个 Agent 产品是否靠谱，可以用这五个问题快速体检：

1. **目标是否清楚？** 它知道成功标准是什么吗？
2. **工具是否够用？** 它能拿到完成任务所需的信息和权限吗？
3. **权限是否分级？** 读、写、删除、发送这些动作有没有边界？
4. **过程是否可见？** 用户能不能看到它做了什么、为什么这么做？
5. **失败是否可恢复？** 出错后能不能撤回、重试、交给人处理？

很多宣传里的 Agent 听起来像“全自动万能助手”，但真正可靠的 Agent 往往很克制：权限清楚、过程透明、关键节点有人确认。

## Agent 的局限

Agent 的问题通常来自“模型能力”和“行动权限”的叠加：

- **会把小错误放大**：第一步理解错了，后面越努力越偏。
- **成本更高**：多轮思考和工具调用会消耗时间和费用。
- **容易卡在现实系统里**：网页改版、接口报错、权限不足，都会让流程中断。
- **安全要求更高**：能操作文件、邮件、数据库，就必须有权限控制和审计记录。
- **评估更难**：聊天 AI 看回答质量，Agent 还要看过程是否正确、是否可追踪、是否可恢复。

所以现阶段最实用的 Agent 形态，不是“完全替代人”，而是“在清晰边界里替人跑腿、整理、检查、执行”。

## 本章小结

- Agent 的本质，是让大模型进入“目标-行动-反馈”的循环。
- 它和普通对话 AI 的区别，不是话术更漂亮，而是能调用工具并推进任务。
- 一个可靠 Agent 至少要有规划、工具调用、观察校验和权限边界。
- 适合交给 Agent 的任务，通常步骤多、可检查、可回退。
- 越靠近真实操作，越要保留人类确认。

## 思考题

1. 你每天有哪些任务不是“难”，而是“步骤多、很琐碎”？如果交给 Agent，它需要哪些工具？
2. 一个能自动发邮件的 Agent，哪些情况下应该停下来让你确认？
3. 如果让 Agent 帮你做一次资料研究，你会要求它展示哪些证据，才愿意相信结论？
