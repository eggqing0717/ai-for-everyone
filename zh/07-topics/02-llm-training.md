# 专题2 大模型训练与微调：模型是怎么从“会续写”变成“会帮忙”的

> 大模型不是一出生就会聊天、写方案、改代码。它通常要经历几次关键训练：先在海量文本里学会语言和知识，再学会听指令，最后学会用人更喜欢、更安全的方式回答。

在第21章里，我们用“大学到研究生”的比喻讲过预训练和微调。这篇专题再往里走一步，但目标不是把你变成训练工程师，而是让你看懂一条主线：

> **预训练让模型会续写，SFT 让模型会听话，对齐训练让模型更好用，部署优化让模型跑得起。**

## 一张总览图

![大模型训练全流程](../assets/images/llm-training-pipeline.svg)

| 阶段 | 解决的问题 | 产物 |
| --- | --- | --- |
| 预训练 | 让模型学会语言规律和通用知识 | 基础模型 |
| 监督微调 SFT | 让模型理解“指令-回答”的形式 | 指令模型 |
| 偏好对齐 RLHF/DPO | 让模型更符合人的偏好和安全要求 | 聊天模型 |
| 部署优化 | 让模型更快、更省、更稳定 | 可上线服务 |

这四步不是每个团队都会从头做。绝大多数人不会自己预训练一个基础模型，而是选择开源或商业基础模型，在上面做微调、检索增强或应用开发。

## 一、预训练：先学会“世界上的话通常怎么说”

预训练是最贵、最基础的一步。它把海量文本切成一个个 token，让模型反复做一件看似简单的事：

> 给定前面的内容，预测下一个 token。

比如看到“杭州今天的天气”，模型要猜后面更可能接“很好”“预报”“怎么样”。只做这件事，为什么能学出那么多能力？因为为了猜准下一个词，模型不得不学习语法、事实、常识、风格、推理模式，甚至代码结构。

### 一个小数字例子：模型到底在算什么？

先看一个极简句子：

> 我 喜欢 吃 苹果

训练时，模型不会一次性“理解整句话”，而是把它拆成一连串预测题：

| 已经看到 | 正确答案 |
| --- | --- |
| 我 | 喜欢 |
| 我 喜欢 | 吃 |
| 我 喜欢 吃 | 苹果 |

假设模型看到“我 喜欢 吃”后，给三个候选词打分：

| 候选下一个词 | 模型认为的概率 |
| --- | --- |
| 苹果 | 0.70 |
| 跑步 | 0.20 |
| 天气 | 0.10 |

正确答案是“苹果”。概率 0.70 说明模型猜得还不错，但还没到完全确信。训练算法会做一件事：**让“苹果”的概率再高一点，让其他错误选项的概率低一点。**

这背后常用的损失函数叫**交叉熵损失**。你不用记公式，只要抓住直觉：

> 正确答案概率越高，损失越小；正确答案概率越低，损失越大。

例如：

| 正确词概率 | 模型表现 | 损失直觉 |
| --- | --- | --- |
| 0.90 | 很有把握猜对 | 小 |
| 0.50 | 有点犹豫 | 中 |
| 0.05 | 几乎猜错 | 大 |

训练就是不断重复这个过程：模型先猜，下一个词揭晓，算出差距，再微调内部参数。亿万次之后，它就不只是记住“苹果”这个词，而是学会了大量语言和世界规律。

### 数据比想象中更重要

预训练不是把互联网一股脑倒进去。数据要经过大量清洗：

- 去掉重复内容，避免模型只是在背诵。
- 过滤低质量网页、垃圾内容和明显有害内容。
- 移除隐私信息，比如身份证号、手机号、邮箱。
- 平衡不同语言、领域、代码和长文本比例。
- 保留高质量资料，让模型学到更稳定的表达和知识结构。

一个容易被忽略的事实是：**数据质量往往比数据数量更影响体验。** 低质量数据越多，模型越容易学到啰嗦、偏见、错误格式和坏习惯。

### 为什么训练这么贵？

大模型训练贵，主要贵在三件事：

| 成本来源 | 为什么贵 |
| --- | --- |
| 参数多 | 模型越大，每次计算要更新的权重越多 |
| 数据多 | 每个 token 都要经过前向计算和反向更新 |
| 集群复杂 | 成千上万张 GPU 要协同工作，通信和稳定性都很难 |

因此，大公司才会投入巨额算力做基础模型。普通开发者真正需要关心的是：如何选一个合适的基础模型，以及如何把它改造成自己的应用能力。

### 参数更新：错误是怎么变成“学习”的？

模型内部有大量参数，可以先把它们理解成一堆旋钮。每个旋钮都会影响模型最后给各个词打多少分。

一次训练大致是这样：

1. **前向计算**：输入“我 喜欢 吃”，模型算出“苹果”的概率是 0.70。
2. **计算损失**：正确答案是“苹果”，概率还不够高，所以有损失。
3. **反向传播**：系统追踪哪些参数让“苹果”分数不够高、让错误词分数偏高。
4. **更新参数**：把相关旋钮轻轻拧一点，让下次更容易猜对。

注意这个“轻轻”很重要。如果一次拧太大，模型可能从一个错误跳到另一个错误；如果拧太小，学习又太慢。所谓**学习率**，就是控制每次拧旋钮幅度的数字。

## 二、SFT：教模型听懂人的指令

预训练后的模型更像“会接着写”的文本机器。你写：

> 翻译成英文：今天天气很好

它未必稳定地给出翻译，可能继续补一段例句，也可能学网页格式继续展开。SFT，也就是监督微调，要解决的就是这个问题：让模型知道人类发出的是任务，而不是让它随便续写。

SFT 数据通常长这样：

```json
{
  "instruction": "将下面这句话翻译成英文",
  "input": "机器学习是人工智能的一个分支。",
  "output": "Machine learning is a branch of artificial intelligence."
}
```

高质量 SFT 数据会覆盖多种任务：

- 问答：准确、直接地回答问题。
- 摘要：抓重点，不乱加信息。
- 写作：按受众、语气、长度生成内容。
- 代码：解释、补全、修改、测试。
- 工具使用：知道什么时候该查资料、算数、调用接口。
- 拒答和安全：遇到不该做的请求时合理拒绝。

SFT 的关键不是“越多越好”，而是“示范要像你希望模型以后表现的样子”。如果训练数据啰嗦，模型就会啰嗦；如果示范里经常编造引用，模型也会学坏。

### SFT 的计算和预训练有什么不同？

SFT 本质上仍然是在做“预测下一个 token”，只是训练材料换成了人类希望它模仿的问答格式。

比如这条样本：

```text
用户：用一句话解释机器学习。
助手：机器学习就是让计算机从数据中总结规律，并用这些规律做预测。
```

训练时，模型通常只对“助手回答”这部分计算损失。也就是说，前面的“用户问题”是条件，后面的“助手回答”是要学习生成的目标。

可以拆成这样：

| 已经看到 | 要预测 |
| --- | --- |
| 用户问题 + 助手： | 机器 |
| 用户问题 + 助手：机器 | 学习 |
| 用户问题 + 助手：机器 学习 | 就是 |
| …… | …… |

这就是为什么 SFT 能改变模型的行为。预训练时，它学的是“互联网上各种文本会怎么接着写”；SFT 时，它学的是“面对用户请求，一个好助手应该怎么接着写”。

再说得更直白一点：

> 预训练教模型补全文本，SFT 教模型补全“有用的回答”。

## 三、LoRA 和 QLoRA：普通团队怎么微调大模型

全参数微调就像把整栋楼重新装修：效果上限高，但非常贵，也容易破坏原来的能力。LoRA 的思路更轻巧：不改主体结构，只在关键位置加一小组可训练的“旁路参数”。

![LoRA 原理示意图](../assets/images/lora-architecture.svg)

可以这样理解：

| 方法 | 像什么 | 优点 | 代价 |
| --- | --- | --- | --- |
| 全参数微调 | 重写整本书 | 改动彻底，上限高 | 显存和数据要求高，容易遗忘旧能力 |
| LoRA | 给书加一叠便签 | 训练参数少，成本低 | 能改的范围有限 |
| QLoRA | 把原书压缩后再贴便签 | 显存更省，个人显卡也能尝试 | 训练和推理配置更讲究 |

LoRA 的好处在于，它冻结原模型的大部分权重，只训练很少一部分新增参数。QLoRA 进一步把基础模型量化到 4-bit，再训练 LoRA 参数。QLoRA 论文显示，这类方法能显著降低显存需求，让更大的模型可以在有限硬件上微调。

### LoRA 的矩阵计算：为什么两个小矩阵能代替一个大改动？

模型里的很多层，本质上是在做这样的计算：

```text
输出 y = W × 输入 x
```

这里的 `x` 是输入向量，`W` 是权重矩阵，`y` 是输出向量。你可以把 `W` 理解成一张很大的转换表：它决定“输入里的哪些信息，应该怎样组合成输出”。

全参数微调会直接改 `W`。LoRA 不这么做，它说：

```text
新的输出 y = W × x + ΔW × x
```

其中 `W` 保持不动，只学习一个小改动 `ΔW`。LoRA 的关键假设是：这个改动不用完整大矩阵表示，可以拆成两个小矩阵相乘：

```text
ΔW = B × A
所以 y = W × x + B × A × x
```

看一个小数字例子。假设输入 `x` 有 4 个数字，输出也有 4 个数字，那么完整的 `W` 是一个 4×4 矩阵，一共有 16 个参数。

如果直接学习 `ΔW`，也要学 16 个参数：

```text
ΔW：4 × 4 = 16 个参数
```

LoRA 选择一个很小的中间维度，比如 `r = 2`：

```text
A：2 × 4 = 8 个参数
B：4 × 2 = 8 个参数
B × A：仍然得到 4 × 4 的改动效果
```

这个玩具例子里参数数量刚好没省，但真实模型的矩阵会大得多。假设一层是 4096×4096：

```text
完整 ΔW：4096 × 4096 = 16,777,216 个参数
LoRA A：16 × 4096 = 65,536 个参数
LoRA B：4096 × 16 = 65,536 个参数
合计：131,072 个参数
```

也就是说，原本要训练 1677 万个参数，现在只训练 13 万个左右。它不是没有代价：`r` 越小，能表达的改动越有限；但好处是成本大幅下降，也更不容易把原模型已有能力破坏掉。

### A 和 B 分别在做什么？

可以把 `A` 和 `B` 看成一个“先压缩、再展开”的过程：

```text
x 原本有 4096 个数字
↓ 乘 A
压缩成 16 个数字
↓ 乘 B
再展开回 4096 个数字
```

这 16 个数字像一组很窄的“调节通道”。模型不能随便大改，只能通过这组通道对原输出做补充。对很多微调任务来说，这已经够用了：你不是要重造一个模型，只是想让它更懂某种格式、语气、行业流程或任务习惯。

第一次尝试微调时，不建议一上来追求大模型和复杂参数。更稳的路线是：

1. 先选一个中小尺寸、中文能力不错的开源模型。
2. 准备几百到几千条高质量任务样例。
3. 用 LoRA 或 QLoRA 跑通完整流程。
4. 用人工测试集检查回答质量。
5. 再决定是否扩大数据、模型或训练轮数。

## 四、RLHF 和 DPO：让模型不只“能答”，还要“答得好”

SFT 教会模型“怎么完成任务”，但很多问题没有唯一标准答案。比如同样是解释一个概念，有的回答准确但生硬，有的回答清楚、友好、有边界。对齐训练要优化的就是这种“人更喜欢哪一个”的差异。

传统 RLHF 大致分三步：

1. 让模型对同一个问题生成多个回答。
2. 请人类标注哪个回答更好，训练一个奖励模型。
3. 用强化学习方法优化模型，让它更倾向于生成高分回答。

RLHF 很有效，但工程复杂、训练不稳定、成本也高。后来 DPO（Direct Preference Optimization）把流程简化了：它直接用“选中的回答”和“没选中的回答”训练模型，不再显式训练奖励模型，也不需要完整强化学习流程。DPO 论文把它描述为一种更稳定、计算更轻的偏好优化方法。

| 方法 | 适合谁 | 特点 |
| --- | --- | --- |
| RLHF | 有大规模标注和训练能力的团队 | 流程成熟，但工程复杂 |
| DPO | 中小团队、研究者、开源微调 | 更简单，接近 SFT 的训练体验 |

对普通读者来说，记住这句话就够了：

> **SFT 让模型知道“该怎么答”，偏好对齐让模型知道“哪种答法更好”。**

### DPO 的小数字例子：把“更喜欢 A”变成训练信号

假设同一个问题有两个回答：

```text
问题：什么是过拟合？
回答 A：过拟合就是模型把训练数据背得太死，换到新数据上表现变差。
回答 B：过拟合是一种机器学习现象。
```

人类标注员选择 A 更好。DPO 要做的事，就是让模型以后更倾向于生成 A，而不是 B。

它会比较两件事：

| 回答 | 当前模型给出的倾向 |
| --- | --- |
| A | 0.55 |
| B | 0.45 |

模型已经稍微偏向 A，但还不够。训练会把 A 的倾向推高，把 B 的倾向压低。下一轮可能变成：

| 回答 | 训练后倾向 |
| --- | --- |
| A | 0.75 |
| B | 0.25 |

DPO 的重点不是让模型死记这一个问题，而是让它从大量“A 比 B 好”的例子里学出偏好：回答要具体、清楚、诚实、有帮助，遇到危险请求要有边界。

和 SFT 相比，DPO 多学了一层东西：

- SFT 看到的是“标准答案长这样”。
- DPO 看到的是“两个答案放一起，人更喜欢哪一个”。

这就像学写作文：只看范文当然有用，但如果老师告诉你“这两篇里哪篇更好，为什么更好”，你会更快抓住好文章的标准。

## 五、部署优化：训练好了，还要跑得起

模型训练完成后，还不能直接高枕无忧。上线服务时最现实的问题是：慢、贵、占显存。

常见优化手段包括：

| 技术 | 解决什么问题 | 直观理解 |
| --- | --- | --- |
| 量化 | 降低模型体积和显存占用 | 把高清大图压缩成更小文件 |
| KV Cache | 避免重复计算已生成内容 | 前面读过的内容记下来 |
| Flash Attention | 加速注意力计算 | 让模型看上下文时少搬数据 |
| 批处理 | 提高吞吐量 | 多个请求拼车处理 |
| 推测解码 | 提升生成速度 | 小模型先打草稿，大模型审核 |

这一步决定了模型能不能真正服务用户。一个实验室里效果不错的模型，如果响应太慢、成本太高、并发太差，也很难变成产品。

### 量化的小数字例子：为什么 4-bit 能省显存？

模型参数本质上是一堆数字。原来每个数字如果用 FP16 保存，通常占 16 bit；如果量化到 4-bit，每个数字只占 4 bit。

```text
16 bit ÷ 4 bit = 4
```

所以，同样数量的参数，理论上体积可以接近缩小到四分之一。

比如一个小模型有 100 亿个参数：

| 保存方式 | 每个参数占用 | 粗略体积 |
| --- | --- | --- |
| FP16 | 2 字节 | 约 20 GB |
| INT8 | 1 字节 | 约 10 GB |
| INT4 | 0.5 字节 | 约 5 GB |

代价是数字会变粗糙。原来一个参数可能是 `0.1372`，量化后可能只能近似成 `0.125`。如果近似误差控制得好，模型质量下降不明显；如果压得太狠，回答质量就会变差。这就是部署优化里永远绕不开的取舍：**更省、更快，通常要用一点精度来换。**

## 六、普通人真正该怎么用这套知识？

你未必需要训练模型，但理解训练流程能帮你做三个判断。

### 1. 判断产品宣传是否靠谱

如果一个产品说“我们有行业专属大模型”，你可以追问：

- 是从零预训练，还是基于开源模型微调？
- 数据来自哪里，是否有授权和脱敏？
- 是只做了提示词，还是做了 SFT、DPO、RAG？
- 有没有专门测试集，效果怎么评估？

问清这些，比听“千亿参数”“行业领先”更有用。

### 2. 判断自己的需求该选哪条路

| 需求 | 更合适的方案 |
| --- | --- |
| 让模型回答公司资料 | 优先 RAG，不一定要微调 |
| 让模型固定一种写作格式 | 少量 SFT 或提示词模板 |
| 让模型掌握行业流程 | SFT + 工具/RAG |
| 让模型语气更符合偏好 | DPO 或高质量偏好样例 |
| 让模型拥有全新基础知识 | 先考虑数据检索，微调不是万能记忆库 |

很多场景下，RAG 比微调更便宜、更可控。微调擅长改变行为模式，不擅长当作实时知识库。

### 3. 判断微调是否成功

不要只看训练 loss。真正要看模型在真实任务上的表现：

- 是否更准确，而不是更会编。
- 是否保持原有通用能力，没有明显退化。
- 是否在边界问题上会承认不知道。
- 是否能稳定遵守格式、语气和流程。
- 是否能通过一组固定测试题反复验证。

## 本章小结

- 预训练让模型通过“预测下一个 token”学会语言、知识和模式。
- 损失函数负责衡量“正确答案概率够不够高”，反向传播负责把错误变成参数更新。
- SFT 仍然是预测下一个 token，只是学习目标换成了“好助手应该怎么回答”。
- LoRA 把大矩阵改动 `ΔW` 拆成 `B × A`，用少量参数实现可控微调。
- RLHF/DPO 用人类偏好继续调整模型，让回答更有帮助、更符合安全边界。
- 量化等部署优化决定模型能不能跑得快、跑得省、跑得稳定。
- 对多数应用来说，微调不是第一选择；提示词、RAG、工具调用和评估体系同样重要。
