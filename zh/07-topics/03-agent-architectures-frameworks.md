# 专题 3：Agent 常用架构与框架，AI 团队是怎样组织起来的

> 一个 Agent 能调用工具，只解决了“它会不会做事”的问题。真正把 Agent 放进产品，还要回答：谁来拆任务？中间结果放在哪里？失败后从哪一步重来？什么时候必须请人确认？这些安排合在一起，就是 Agent 的架构。

## 先分清三个容易混在一起的词

| 名称 | 它回答的问题 | 可以把它理解成 |
| --- | --- | --- |
| 模型 | 谁来理解、判断和生成内容？ | 大脑 |
| 架构 | 任务按什么方式流动和分工？ | 组织结构与办事流程 |
| 框架 | 用什么现成工具把架构做出来？ | 搭建这套组织的软件工具箱 |

同一个模型可以放进不同架构，同一种架构也能用不同框架实现。用了某个 Agent 框架，不等于系统就自动拥有了可靠的规划能力；框架只是帮开发者管理循环、状态、工具、交接和运行记录。

下面都用同一个任务来观察不同架构：

> “收集三家同类产品最近一年的公开信息，比较价格和功能，写一份有来源的报告。”

## 架构一：单 Agent 工具循环

这是最基础的 Agent 架构。一个模型既负责判断下一步，也负责选择工具和整理结果。

```text
用户目标
  → Agent 判断下一步
  → 调用搜索或浏览器
  → 读取工具返回结果
  → 更新当前状态
  → 再次判断下一步
  → ……直到生成报告
```

每轮循环都可以写成一个简单的状态变化：

```text
新状态 = 旧状态 + 本轮动作 + 工具返回结果
```

例如，开始时的状态只有：

```text
目标：比较 A、B、C 三个产品
已完成：0 家
已找到来源：0 个
```

搜索完产品 A 后，状态可能变成：

```text
目标：比较 A、B、C 三个产品
已完成：A
待处理：B、C
已找到来源：3 个
异常：A 的企业版价格未公开
```

Agent 下一轮看到的不是一句孤立的工具回复，而是“任务走到了哪里”。这就是状态的作用。

**适合：** 步骤不多、允许灵活探索、出错后容易重试的任务。

**风险：** 模型每轮都在临场决定，可能重复搜索、忘记约束，或者在错误方向上越走越远。

## 架构二：带路由的固定工作流

并不是每一步都需要 AI 自由决定。很多系统会先画好主流程，只在需要理解和判断的节点使用模型。

```text
接收任务
  → 判断产品名称是否齐全
  → 并行收集三家资料
  → 检查来源数量
      ├─ 不足：继续搜索
      └─ 足够：进入比较
  → 生成报告
  → 人工确认
```

这里的“路由”就像路口的指示牌。假设规定每个产品至少要有 2 个可靠来源，检查结果为：

```text
A：3 个
B：1 个
C：2 个
```

那么路由器不会让整个任务重做，只把 B 送回搜索节点。模型负责理解资料，程序负责执行明确规则。

**适合：** 流程比较稳定、需要审计、失败后要精确恢复的业务。

**风险：** 流程画得太死，会应付不了事先没想到的情况；分支太多，又会变得难以维护。

这是当前很实用的一种思路：**用确定的流程管住边界，把有限的自主权留给真正需要判断的步骤。**

## 架构三：规划者—执行者

复杂任务很难靠“想到哪做到哪”完成，于是可以把规划和执行分开。

```text
规划者：把目标拆成任务清单
   ↓
执行者：逐项调用工具
   ↓
检查者：核对结果是否满足要求
   ├─ 不满足：把问题交回规划者调整
   └─ 满足：汇总输出
```

规划者可能先生成这样的任务表：

| 编号 | 子任务 | 完成条件 | 状态 |
| --- | --- | --- | --- |
| 1 | 收集三家官网价格 | 每家至少一个官方来源 | 进行中 |
| 2 | 整理核心功能 | 使用统一的功能分类 | 未开始 |
| 3 | 查找一年内的重要更新 | 记录发布日期和来源 | 未开始 |
| 4 | 比较并写报告 | 每个关键结论可追溯 | 未开始 |

执行者不必重新理解整个目标，只处理当前子任务。检查者发现“产品 B 没有公开企业价”时，可以把结果标成“需询价”，而不是编造一个数字。

**适合：** 子任务多、完成标准能说清、执行中需要调整计划的任务。

**风险：** 规划本身也可能错；计划写得很漂亮，不代表每一步都能执行。规划、执行、检查还会增加模型调用次数。

假设一次直接回答要调用模型 2 次，而规划、4 个子任务和最终检查共调用 6 次；若每次平均处理 3,000 个 token，模型处理量会从约 6,000 个增加到约 18,000 个。架构换来了可控性，也带来了时间和成本。

## 架构四：主管—专家，多 Agent 分工

当任务需要明显不同的专业能力时，可以让多个 Agent 各司其职。

```text
主管 Agent
  ├─ 搜索 Agent：寻找和筛选来源
  ├─ 数据 Agent：提取价格与功能
  ├─ 分析 Agent：进行横向比较
  └─ 写作 Agent：形成最终报告
```

主管负责分派任务、接收结果和决定是否返工。专家 Agent 可以拥有不同的提示词、工具和权限：搜索 Agent 能访问网页但不能发邮件；写作 Agent 能读取整理后的资料，却不能修改原始数据。

多 Agent 的价值不在于“多开几个模型就更聪明”，而在于：

- 每个角色的目标更窄，更容易约束和测试。
- 工具与权限可以按职责隔离。
- 不同子任务可以并行执行。
- 某个角色可以替换，而不必重做整个系统。

它的代价也很真实。Agent 之间通过文字或结构化数据交接，信息可能丢失或被误解。假设搜索 Agent 找到 20 条资料，却只向分析 Agent 传递 8 条摘要，后者的判断上限就由这 8 条摘要决定，而不是由原始的 20 条资料决定。

**适合：** 专业分工明确、子任务可独立验证、权限需要隔离的复杂任务。

**不适合：** 本来一个 Agent 三步就能做完的事情。角色越多，沟通成本和排错难度越高。

## 架构五：并行、评审与反思

有些问题没有唯一正确答案，可以让多个执行者独立作答，再由评审者比较。

```text
同一个问题
  ├─ 执行者 A：从价格角度分析
  ├─ 执行者 B：从功能角度分析
  └─ 执行者 C：从用户口碑角度分析
          ↓
      评审者合并、查冲突、要求补证据
```

另一种做法是“生成—检查—修改”：

```text
初稿 → 检查事实和来源 → 指出问题 → 修改初稿
```

这类架构能减少单次回答的偶然错误，但不能保证真相自动出现。如果三个执行者都引用了同一条错误资料，投票结果仍然会错。因此，评审规则应检查证据，而不只是数一数谁赞成。

**适合：** 研究、方案比较、内容审核、代码审查等可以交叉验证的任务。

**风险：** 调用次数多，评审者也会判断失误，还可能陷入反复修改却没有明显进步的循环。

## 真实产品通常是混合架构

上面五种不是互斥的产品类别，更像五种积木。一个可靠的报告 Agent 可能这样组合：

1. 固定工作流规定“收集—检查—分析—确认”的主路径。
2. 规划者把模糊目标拆成可检查的子任务。
3. 三个搜索执行者并行收集不同产品的信息。
4. 路由器把来源不足的产品送回搜索。
5. 写作 Agent 只读取核验后的数据。
6. 对外发布前由人确认。

越接近真实业务，越不能只画“几个 Agent 互相聊天”的图。还要补上状态存储、权限控制、失败重试、人工确认、运行记录和质量评估。

## 常见框架在解决什么问题

框架变化很快，下面不是排行榜，而是一张“各自主要抽象什么”的地图。

| 框架 | 核心思路 | 比较适合 |
| --- | --- | --- |
| **LangChain / LangGraph** | LangChain 提供模型、工具和 Agent 抽象；LangGraph 用“状态、节点、边”描述可循环、可恢复的流程 | 需要精细控制状态、分支、长任务和人工介入 |
| **OpenAI Agents SDK** | 用 Agent、工具、交接、护栏、会话和运行追踪组成较轻量的 Agent 运行时 | 使用 OpenAI 模型，希望较快搭建单 Agent 或专家交接 |
| **Microsoft AutoGen** | 把 Agent 看成可收发消息的独立单元，支持对话式多 Agent 和事件驱动系统 | 多 Agent 协作、研究实验、需要灵活通信的系统 |
| **CrewAI** | 用 Crew、角色、任务和流程模拟一支分工团队，同时用 Flow 管理确定的业务流程 | 角色分工直观、团队式协作和业务自动化 |
| **Semantic Kernel** | 把模型、插件、记忆、流程和 Agent 接入应用，提供顺序、并行、交接、群聊等编排模式 | 已在微软或 .NET 技术体系中，需要企业应用集成 |

以 LangGraph 为例，一次运行可以被表示为：

```text
状态 State：保存目标、资料、错误和进度
节点 Node：搜索、分析、写作、人工确认
边 Edge：规定下一步去哪里
```

而 OpenAI Agents SDK 更强调 Agent 运行循环以及专家之间的交接。例如分诊 Agent 判断用户要退款后，把任务和必要上下文交给退款 Agent。AutoGen 和 CrewAI 则更容易用“几个角色怎样协作”的方式思考系统。

这些框架能减少重复开发，但都不能替开发者决定：

- 哪些步骤应该交给模型，哪些应该写成确定规则。
- 工具返回的数据是否可信。
- 一个 Agent 可以读、写、删除哪些东西。
- 失败时应该重试、换路，还是停下来找人。
- 用什么测试集判断系统真的变好了。

## 选择架构时，先别急着选择框架

可以先问六个问题：

| 问题 | 它影响的设计 |
| --- | --- |
| 任务路径稳定吗？ | 稳定就多用固定工作流，不稳定才增加自主规划 |
| 中间结果要保存吗？ | 长任务需要明确的状态和恢复机制 |
| 子任务能独立完成吗？ | 能独立完成才适合并行或多 Agent |
| 不同角色需要不同权限吗？ | 需要时可按 Agent 隔离工具和数据 |
| 错误后果严重吗？ | 后果越严重，越需要规则校验和人工确认 |
| 怎样知道它做对了？ | 先定义完成标准、测试样例和运行记录 |

一个很实用的原则是：**从能完成任务的最简单架构开始。** 单 Agent 足够，就不必先组织一支 AI 团队；固定流程能解决，就不要把每个路口都交给模型临场决定。只有当任务确实需要并行、专业分工或动态调整时，再增加角色和层次。

## 看懂 Agent 架构的三条主线

以后再看到新的 Agent 产品或框架，可以先不管名字，沿着三条线追问：

1. **控制流：** 谁决定下一步？是固定规则、一个规划者，还是多个 Agent 协商？
2. **数据流：** 原始资料、中间结果和最终答案怎样传递？传的是全文、摘要，还是结构化字段？
3. **权限流：** 谁能调用哪些工具？哪些动作能自动执行，哪些必须由人批准？

只要这三条线能说清，一个复杂架构就不再神秘。反过来，如果产品只展示“多个智能体正在协作”，却说不清状态、证据、权限和失败处理，那么那张热闹的架构图并不能证明系统可靠。

## 本章小结

- 模型是大脑，架构是任务的组织方式，框架是实现架构的工具箱。
- 常见架构包括单 Agent 循环、固定工作流、规划者—执行者、主管—专家，以及并行评审。
- 多 Agent 主要解决分工、并行和权限隔离问题，不会因为 Agent 数量增加就自动更聪明。
- 真实产品常用混合架构：确定流程管边界，Agent 处理需要理解与判断的节点。
- 选择框架之前，应先画清控制流、数据流和权限流。
- 架构越复杂，模型调用、通信损耗、运行成本和排错难度通常越高。

## 思考题

1. 如果让 Agent 帮你规划一次旅行，哪些步骤适合固定工作流，哪些步骤需要模型判断？
2. 一个“搜索 Agent”把十篇网页压缩成五句话交给“写作 Agent”，可能丢掉哪些重要信息？
3. 在付款、发送邮件和修改数据库这三类动作中，你会怎样安排权限和人工确认？
